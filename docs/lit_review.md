# Literature Synthesis — Project 5: De Bruijn Graph Genome Assembly

This review is written after reading three foundational papers in the `literature/` folder. Page numbers and quotations refer to those PDFs.

## 1. The problem

Shotgun sequencing fragments a genome into millions of short overlapping reads and the assembly problem is to reconstruct the original sequence from these fragments. Two algorithmic paradigms have competed for this task. The overlap layout consensus approach treats each read as a node and each pairwise overlap as an edge, then seeks a Hamiltonian path visiting every node exactly once. Hamiltonian path is NP hard in general and becomes computationally painful at short read volumes, a point Compeau Pevzner and Tesler 2011 make bluntly on page 2 of their Nature Biotechnology primer: "computer scientists would see this as an ominous development, because finding a Hamiltonian path in a large graph is a notoriously intractable problem." The de Bruijn graph approach, introduced for assembly by Idury and Waterman 1995 and crystallized by Pevzner Tang and Waterman 2001, replaces the Hamiltonian formulation with an Eulerian path problem, which Euler himself reduced in 1735 to a simple degree condition and which Hierholzer 1873 solved in linear time.

## 2. The de Bruijn graph construction

Given a set of reads and a chosen integer k, the de Bruijn graph is defined so that every observed k-mer corresponds to an edge from its k minus one prefix to its k minus one suffix. Equivalently, nodes are the distinct k minus one-mers appearing as prefixes or suffixes of observed k-mers, and each observed k-mer contributes one directed edge between the two. An ordering of reads that reconstructs the original sequence then corresponds to an Eulerian path in this graph, a walk traversing every edge exactly once. Pevzner Tang and Waterman 2001 formalize this on page 9749 by defining the de Bruijn graph G of S equals (s1 ... sn) with vertex set S sub (l minus 1), "the set of all (l minus 1)-tuples from S," and edges added as follows: an (l minus 1)-tuple v in S sub (l minus 1) is joined by a directed edge to w in S sub (l minus 1) if S sub l contains an l-tuple for which the first l minus 1 nucleotides coincide with v and the last l minus 1 nucleotides coincide with w. A path visiting every edge exactly once is by construction an assembly of the reads.

## 3. Why Eulerian beats Hamiltonian

Compeau Pevzner Tesler 2011 demonstrate the Eulerian reformulation with a clean five-read toy example on page 3 (Figure 3). When the five reads ATG, TGG, TGC, GTG, GGC are drawn as nodes of an overlap graph and connected by overlap edges, finding the assembly requires a Hamiltonian path (NP hard). When the same reads are drawn as edges of a de Bruijn graph over two-letter nodes, finding the assembly requires an Eulerian path, which any modern graph library solves in linear time. The difference at scale is decisive: mammalian assemblies involve billions of reads, and the overlap approach does not scale while the de Bruijn approach does. Pevzner Tang and Waterman 2001 state this directly on page 9749: "Below we demonstrate that the repeat problem largely disappears if one uses DB sequencing instead of SBH. Although it sounds counterintuitive, the computational problem of fragment assembly with 10 sup 6 reads is easier to solve than the SBH problem with a few thousand probes."

## 4. The four hidden assumptions

Every practical implementation must confront the gap between the clean theory and messy real data. Compeau Pevzner and Tesler 2011 enumerate in Box 2 (page 5) four assumptions that are false for any next-generation sequencing data: reads are not perfect (they contain errors), reads do not come from one strand (they come from both, requiring reverse-complement handling), reads are not all the same length, and reads do not cover the genome uniformly. Each of these assumptions corresponds to an engineering concern the implementation must address, and each has an established mitigation technique that these three papers jointly develop.

## 5. Tips, bubbles, and erroneous connections

Sequencing errors corrupt the de Bruijn graph in three topologically distinct ways. Zerbino and Birney 2008 identify them by name in the Velvet paper on page 823: "Erroneous data create three types of structures: 'tips' due to errors at the edges of reads, 'bulges' due to internal read errors or to nearby tips connecting, and erroneous connections due to cloning errors or to distant merging tips." A tip is a chain of nodes that is disconnected at one end, that is, a short dead-end branch. A bubble is a pair of internally disjoint paths with the same start and end node, formed when a single-nucleotide error inside a read produces a parallel path differing by one or two characters. An erroneous connection is a spurious low-coverage edge between otherwise disconnected regions, created when a chimeric read or a cross-library contamination introduces an overlap that the genome does not actually contain.

The three features are removed consecutively in Velvet. Tips are clipped first with two criteria: a length criterion and a minority-count criterion. From page 823: "A tip will only be removed if it is shorter than 2k. This arbitrary cutoff length was chosen because it is greater than the length in k-mers of an individual very short read." The minority count requires that the arc leading out of the tip at the junction has multiplicity strictly less than at least one alternative at that junction, so genuine low-coverage sequence is not destroyed. Bubbles are then removed with the "Tour Bus" algorithm, a Dijkstra-like breadth-first search from every node that compares the sequences along any two paths arriving at the same node and merges them into the higher-coverage path if the sequences are similar enough. Finally erroneous connections are removed with a simple coverage cutoff set interactively from a coverage histogram. Our implementation reproduces this three-step pipeline in simplified form: `remove_tips` for Velvet step one, `pop_bubbles` with a coverage ratio criterion for step two, and `remove_low_coverage_edges` for step three.

## 6. The twin-node trick and the oddness of k

Velvet introduces a specific convention for handling the double-stranded nature of DNA that our implementation follows. On page 822 Zerbino and Birney state: "Each node N is attached to a twin node N̄, which represents the reverse series of reverse complement k-mers. This ensures that overlaps between reads from opposite strands are taken into account. Note that the nodes attached to a node and its twin do not need to be reverse complements of each other. The union of a node N and its twin N̄ is called a 'block.'" Our `canonical` function implements the block identity by always collapsing a k-mer and its reverse complement to the lexicographically smaller of the two. Zerbino and Birney note a subtle but important constraint on page 822: "To ensure that each k-mer cannot be its own reverse complement, k must be odd." A palindromic k-mer would be its own twin and would create self-loops that corrupt the algebra. Our default tests and demo use odd k (3, 5, 15, 21, 25) in compliance with this rule.

## 7. The Eulerian path algorithm

Hierholzer's 1873 algorithm for Eulerian paths runs in O(E) time and our `find_eulerian_path` is a direct transcription of it. Compeau Pevzner and Tesler 2011 describe the algorithm intuitively on page 3 using an ant metaphor: "An ant walks along the edges of a Eulerian graph, with the intention of traversing each edge exactly once. Each time it enters a vertex it has not seen before, it arbitrarily picks an unused outgoing edge and follows it. Sooner or later the ant gets stuck — not because it has traversed all edges in the graph, but rather because it has entered a vertex and found no unused outgoing edges." The fix is to splice in cycles: whenever the ant gets stuck, find a vertex on the current walk that still has unused outgoing edges, walk a cycle from there back to itself, and splice that cycle into the walk. Our implementation uses the iterative stack-based variant of this, which is cleaner than the recursive splice.

The existence condition is also well known and we check it directly. A connected graph has an Eulerian path if and only if exactly zero or exactly two vertices have unbalanced in-degree and out-degree, with the two unbalanced ones (if they exist) having an excess of exactly one outgoing and one incoming edge respectively. When this condition fails, `find_eulerian_path` returns None and the caller falls back to `extract_contigs`, the maximal non-branching path decomposition.

## 8. Contigs and the maximal non-branching path decomposition

When no global Eulerian path exists — which is the case for almost every real assembly due to repeats and gaps — the graph is decomposed into maximal non-branching paths called contigs. A non-branching path is a path where every internal node has in-degree one and out-degree one. Zerbino and Birney call these "chains" and explain the simplification on page 822: "Whenever a node A has only one outgoing arc that points to another node B that has only one ingoing arc, the two nodes (and their twins) are merged. Iteratively, chains of blocks are collapsed into single blocks." Our `extract_contigs` implements this compression directly: we walk from every branching or terminal node along the unique successor as long as the path is non-branching, emit the concatenated sequence as a contig, and move on. The result is a set of unambiguously assembled DNA strings whose length distribution is summarized by the N50 statistic, the length N such that contigs of length at least N cover 50 percent of the total assembly.

## 9. Repeats, tangles, and the Eulerian superpath problem

Repeats are the fundamental obstacle that no amount of k-mer counting resolves. When the same l-tuple occurs in two genomic locations the de Bruijn graph collapses it to a single edge with multiple entrances and exits, creating ambiguity that an Eulerian path picks up arbitrarily. Pevzner Tang and Waterman 2001 define a repeat precisely on page 9751 as "a path v1 ... vn in the de Bruijn graph with indegree(v1) greater than 1, outdegree(vn) greater than 1, and indegree(v1) equals outdegree(vi) equals 1 for 1 less than or equal to i less than or equal to n minus 1." A repeat is called a tangle if no read path crosses from a single entrance to a single exit and the graph alone does not determine the pairing. Their solution is the Eulerian Superpath Problem: find an Eulerian path that respects a collection of read-paths as subpaths. They solve it by two graph transformations called x,y-detachment and x-cut that incrementally reduce the superpath problem to an ordinary Eulerian path problem while preserving all consistent pairings. We do not implement the full Eulerian Superpath transformations in this project (they belong to a later generation of assembler), but we do note where our implementation would fail on a repeat-rich assembly: `find_eulerian_path` picks an arbitrary pairing at each branching node, which is correct on single-copy regions and wrong on repeats. Our contig extraction falls back to non-branching paths exactly where repeats create branch nodes.

## 10. Error correction and the spectral approach

The other large open problem is error correction. Pevzner Tang and Waterman 2001 introduce a spectral approach on pages 9750 to 9751: "A k-mer is called solid if it belongs to more than M reads, and weak otherwise, where M is a threshold that reflects coverage." The Error Correction Problem is to introduce a minimum number of nucleotide substitutions in the reads that makes every k-mer solid. They report on page 9751 that the simple procedure already eliminates 86.5 percent of errors, while EULER's more involved approach eliminates 97.7 percent of errors, transforming the original sequencing data with 4.8 errors per read on average into almost error-free data with 0.11 errors per read on average. Our implementation handles error correction implicitly by removing low-coverage edges and tips, a simpler but topologically equivalent operation: spurious k-mers produce tips and low-coverage edges, and removing them is equivalent to erasing the erroneous k-mers from the spectrum.

## 11. Choice of k

Small k gives a well connected graph but collapses repeats and inflates apparent coverage of any matching sequence; large k resolves repeats but fragments the graph into many small components where reads do not overlap by k. Zerbino and Birney note on page 822: "This variable k is limited on the upper side by the length of the reads being hashed, to allow for a small amount of overlap, usually k equals 21 for 25 to 35-bp reads." Figure 3 of the Velvet paper shows N50 as a function of coverage for four species (E. coli, S. cerevisiae, C. elegans, H. sapiens) at fixed k equals 31 and reports N50 around 50 kb on E. coli at 50x perfect coverage. In practice assemblers run several values of k and pick the best by N50, or use multi-k strategies as in SPAdes.

## 12. K-mer counting at scale

For small pedagogical datasets a Python dict mapping k-mer strings to integer counts is adequate, which is what `count_kmers` does. For mammalian-scale data the dict is replaced by specialized structures. Marcais and Kingsford 2011 introduced Jellyfish, which uses a lock-free compare-and-swap hash table on 2-bit-encoded k-mers and scales to tens of billions of k-mers on a workstation. Bloom-filter and count-min-sketch variants (Cormode and Muthukrishnan 2005 for the CM sketch itself, Melsted and Pritchard 2011 for BFCounter) give approximate counts in sub-linear space, at the cost of one-sided error. Our `CountMinSketch` class is included as a pedagogical demonstration of the space-accuracy tradeoff; it is not used in the main assembly pipeline but gives correct over-estimates that illustrate the technique.
